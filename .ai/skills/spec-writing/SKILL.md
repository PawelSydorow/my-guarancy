# Spec Writing

Use this skill when a task needs a spec, an architecture note, or a review of an existing spec.

## Workflow

1. Write a small skeleton first.
2. Capture open questions before filling in details.
3. Validate the spec against the current repo layout, existing modules, and test strategy.
4. Break the work into phases that can be implemented and verified independently.
5. Keep the spec short enough to be useful during implementation.

## Good Specs Include

- Problem statement
- Scope
- Data or state changes
- UI or API behavior
- Test plan
- Risks and edge cases

## Rules

- Prefer concrete requirements over vague goals
- Avoid copying irrelevant parent-project patterns
- Keep terminology consistent with this repo
- If a spec becomes too large, split it into phased follow-up docs
